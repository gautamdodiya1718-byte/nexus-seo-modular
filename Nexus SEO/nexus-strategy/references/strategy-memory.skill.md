# SKILL: memory/strategy-memory.skill
# SYSTEM: Nexus — Modular AI-Powered SEO Operating System
# VERSION: 1.0.0
# LAYER: Memory / Strategy Intelligence
# EXECUTION PRIORITY: FOUNDATIONAL — Called at the start of every session to surface active roadmap progress, and at the end of every session to persist all strategic decisions, recommendations, and execution updates.
# POSITION: Strategy continuity layer. The single source of truth for what was planned, what was executed, what was decided, and what the strategic trajectory of the project looks like across all sessions.

## ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
## SECTION 1 — ROLE
## ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

This skill is the **strategic continuity and execution accountability layer** of the Nexus memory system.

Analysis without continuity is noise. A system that produces a brilliant roadmap in Session 1 and then forgets it existed in Session 2 is not a system — it is a series of disconnected tools. A system that logs strategic recommendations but never tracks whether they were acted on or what they produced is a recommendation generator, not a strategy engine.

This skill is what converts Nexus from a collection of analytical tools into a persistent strategic operating system. Every roadmap generated by `roadmap-engine.skill` is versioned and stored here. Every strategic decision made by `strategist-ai.skill` is logged here with its rationale, its source, and its session context. Every recommendation that flows from the system is tracked here for follow-through. Every execution update — a pending item moving to in-progress, a draft becoming published, a roadmap item being skipped — is recorded here and surfaced at the next session start.

When a user opens a new Nexus session, this skill is what makes the system say: "Here is where you are. Here is what was planned. Here is what was executed. Here is what needs to happen next." That context is not re-analyzed from scratch — it is retrieved from this persistent memory layer and used to orient every downstream skill for the session.

This is the layer that turns Nexus from a tool into a system.

### Primary Responsibilities

| Responsibility                          | Description                                                                                                            |
|-----------------------------------------|------------------------------------------------------------------------------------------------------------------------|
| **Roadmap Storage and Versioning**      | Store every roadmap produced by `roadmap-engine.skill` with version numbers; archive superseded versions              |
| **Roadmap Confirmation**                | Require user confirmation before activating a new roadmap as the project's active plan                               |
| **Execution Progress Tracking**         | Track every roadmap item through its lifecycle: pending → in-progress → published / skipped                          |
| **Execution Metrics Maintenance**       | Maintain real-time counts of published, in-progress, pending, and skipped roadmap items                              |
| **Strategic Decision Logging**          | Store every strategic decision from `strategist-ai.skill` with decision text, rationale, source, and session context |
| **Recommendation Tracking**             | Log all system recommendations; track whether they were acted on and what the outcome was                            |
| **Conflict Detection**                  | Before storing a new strategic decision, check whether it contradicts a prior decision in the same project            |
| **Session Continuity**                  | At session start, surface the active roadmap progress and next recommended action                                     |
| **Stagnation Detection**                | Detect when no execution progress has been made across multiple sessions and alert                                    |
| **Execution Gap Detection**             | Identify roadmap items where in-progress status was set but no published status followed                             |
| **Strategy Evolution Tracking**         | Record how the strategy has evolved across roadmap versions — what changed and why                                    |
| **Recommendation Effectiveness Tracking**| Track which recommendations produced positive outcomes when acted on                                               |

### What This Skill Is NOT

- It is **not** a project management tool. It stores strategic records — not sprint tickets, assignments, or calendar events.
- It is **not** a content tracker. Content records live in `content-memory.skill`; this skill stores the roadmap items that reference content.
- It is **not** a keyword manager. Keyword status lives in `keyword-memory.skill`; this skill stores the strategic plan that assigns those keywords to content.
- It is **not** a reporting tool. It provides raw strategic history data — `strategist-ai.skill` converts that data into human-readable strategic summaries.
- It is **not** a real-time performance dashboard. It stores execution state based on user-reported updates and pipeline confirmations — not live analytics pulls.

### The Defining Standard of This Skill

> A strategy that is not tracked is not a strategy. It is a plan that was made and forgotten.
>
> This skill exists to ensure that never happens.
>
> Every decision has a record. Every recommendation has a follow-up.
> Every roadmap item has a status. Every session builds on the last.
>
> What was planned is visible. What was executed is verified.
> What was recommended is tracked. What produced results is remembered.
>
> This is the memory that makes the strategy real.

### Status Enum Definitions — Roadmap Items

| Status          | Definition                                                                                    | Transition From             |
|-----------------|-----------------------------------------------------------------------------------------------|-----------------------------|
| **PENDING**     | Item is scheduled; content creation has not begun                                            | Default at roadmap creation |
| **IN-PROGRESS** | Content creation is actively underway; draft exists                                          | From PENDING                |
| **PUBLISHED**   | Content is live; `content_url` is populated                                                  | From IN-PROGRESS            |
| **SKIPPED**     | Item was explicitly removed from execution without publishing                                | From PENDING or IN-PROGRESS |
| **BLOCKED**     | Item cannot proceed until a dependency item is published                                     | From PENDING                |

## SECTION 5 — STEP-BY-STEP LOGIC
## ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

### ▸ STEP 1 — ROADMAP STORAGE

Receive and store a complete roadmap from `roadmap-engine.skill`.

**Phase 1A — Version Assignment**

1. Query the roadmap index for this project: `strategy:roadmaps:[project_id]`.
2. Identify the highest existing version number.
3. Assign the new roadmap version = highest existing + 1. If no prior roadmap exists → version = 1.
4. Generate a new `roadmap_id` (UUID).
5. Set `status` = DRAFT (not ACTIVE — user confirmation is required before activation).

**Phase 1B — Item Registration**

For every item in the incoming roadmap:
1. Generate a new `item_id` (UUID) for each item.
2. Assign `roadmap_id` reference.
3. Assign `sequence_number` in the order received.
4. Set `status` = PENDING for all items.
5. Store `phase`, `month`, `topic`, `target_keyword`, `intent`, `cluster`, `priority`, `item_type`, `depends_on`, `rationale`.
6. Initialize all optional fields (content_url, content_id, skipped_reason, etc.) to null.

**Phase 1C — Prior Roadmap Handling**

If an ACTIVE roadmap already exists for this project when a new roadmap is stored:

1. Do NOT archive the prior roadmap automatically.
2. Store the new roadmap as DRAFT.
3. Surface a notice:

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
NEW ROADMAP DETECTED
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
A new roadmap (v[N]) has been generated.
Your current active roadmap is v[N-1].

Current v[N-1] status:
  Published: [N] | In Progress: [N] | Pending: [N] | Skipped: [N]

Options:
[A] Activate new roadmap (v[N]) — archive v[N-1]
[K] Keep current roadmap (v[N-1]) active — save new as draft
[M] Merge pending items from v[N] into v[N-1]

What would you like to do?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**Phase 1D — Prior Roadmap Archiving (Option A selected)**

If the user chooses to activate the new roadmap:
1. Set the prior roadmap's `status` = ARCHIVED.
2. Set the prior roadmap's `archived_date` = current timestamp.
3. Set the prior roadmap's `superseded_by` = new roadmap_id.
4. Set the new roadmap's `supersedes` = prior roadmap_id.
5. Proceed to Step 2 for confirmation of the new roadmap.

**Phase 1E — Merge Protocol (Option M selected)**

If the user chooses to merge:
1. Extract all PENDING items from the new roadmap that are NOT already in the active roadmap (by target_keyword or topic match).
2. Add these new items to the active roadmap.
3. Increment the active roadmap's `total_items` count.
4. Log: `"ROADMAP MERGED: [N] new items from v[new version] added to active v[current version]."`
5. Archive the new roadmap as MERGED (not a full ARCHIVED or ACTIVE state): status = ARCHIVED + notes = "Merged into v[current version]."

**Phase 1F — Storage Key Format**

```
Storage Keys:
  strategy:roadmap:[project_id]:[roadmap_id]        → Full roadmap record
  strategy:roadmap:item:[roadmap_id]:[item_id]       → Individual item record
  strategy:roadmaps:[project_id]                     → Index of all roadmap_ids (ordered by version)
  strategy:active:[project_id]                       → Points to the active roadmap_id
  strategy:progress:[project_id]                     → Execution progress record
  strategy:decisions:[project_id]                    → List of all decision_ids for project
  strategy:decision:[project_id]:[decision_id]       → Full decision record
  strategy:recommendations:[project_id]              → List of all recommendation_ids
  strategy:recommendation:[project_id]:[recommendation_id] → Full recommendation record
```

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**Phase 2B — Confirmation Outcomes**

| Response | Action                                                                                           |
|----------|--------------------------------------------------------------------------------------------------|
| Y        | Set roadmap `status` = ACTIVE. Set `confirmed_by_user` = true. Record `confirmed_date` and `confirmation_session`. Update `strategy:active:[project_id]`. Update execution progress record. |
| N        | Set roadmap `status` = DRAFT. Do not update `strategy:active:[project_id]`. Flag: "Roadmap saved as draft — not active. Prior roadmap (if any) remains active." |
| E        | Return roadmap to `roadmap-engine.skill` for modification. Store as DRAFT. Set a flag `awaiting_edit = true`. |

**Phase 2C — Edit Handling**

When E is selected:
1. The user specifies what to change (add/remove items, reorder, change priorities).
2. The changes are passed back to `roadmap-engine.skill` for re-sequencing.
3. A revised roadmap is returned and this confirmation step is run again.
4. The edit loop may occur up to 3 times before the system flags: "Multiple edit cycles detected — consider accepting the current roadmap and making manual adjustments."

### ▸ STEP 3 — PROGRESS TRACKING

Accept and record execution status updates for roadmap items.

**Phase 3A — Status Update Acceptance**

Status updates are received from three sources:

| Source                                | Update Type                                         |
|---------------------------------------|-----------------------------------------------------|
| `content-generation-engine.skill`    | Automatic: PENDING → IN-PROGRESS when generation starts |
| `content-memory.skill`               | Automatic: IN-PROGRESS → PUBLISHED when content confirmed live |
| User input                            | Manual: any update including SKIPPED, BLOCKED, back to PENDING |

**Phase 3B — Status Transition Validation**

Apply the same transition logic as `keyword-memory.skill`:

```
VALID STATUS TRANSITIONS FOR ROADMAP ITEMS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
PENDING → IN-PROGRESS (content creation begins)
PENDING → SKIPPED (item removed from execution)
PENDING → BLOCKED (unmet dependency detected)

IN-PROGRESS → PUBLISHED (content confirmed live)
IN-PROGRESS → SKIPPED (item abandoned)
IN-PROGRESS → PENDING (writing paused — not abandoned)

BLOCKED → PENDING (dependency resolved)
BLOCKED → SKIPPED (item removed despite blockage)

PUBLISHED → [terminal — no further transitions]
SKIPPED → PENDING (reinstatement — allowed)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**Phase 3C — PUBLISHED Status Requirements**

When an item moves to PUBLISHED:
1. Require `content_url` — no PUBLISHED status without a URL.
2. Optionally accept `content_id` (from content-memory record).
3. Record `actual_month_published` if it differs from the planned `month`.
4. If `actual_month_published` > `month` → log a delivery delay: `"Item #[N] published [M] months after planned month [P]."`

**Phase 3D — SKIPPED Status Requirements**

When an item is marked SKIPPED:
1. Require `skipped_reason` — a specific reason, not a vague dismissal.
2. If `skipped_reason` is null → prompt: "Why was this item skipped? (Reason required for strategic record.)"
3. Acceptable reasons: `"SERP too competitive at this time"`, `"Redirected resources to [other item]"`, `"Topic no longer aligned with strategy"`, `"Content created in different format"`, etc.
4. Unacceptable: `"Skipped"` alone, `"N/A"`, `"Unknown"` — these are rejected.

**Phase 3E — Dependency Check on PUBLISHED**

When an item is marked PUBLISHED:
1. Check if any BLOCKED items list this item in their `depends_on`.
2. If yes → automatically transition those items from BLOCKED → PENDING.
3. Notify: `"Item #[N] has been published — [M] dependent items have been unblocked: [list of item topics]."`

### ▸ STEP 4 — EXECUTION METRICS

Maintain the execution progress record in real time with accurate counts.

**Phase 4A — Metric Recalculation**

After every status update, recalculate all execution metrics:

```
METRIC RECALCULATION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
published_count   = COUNT(items WHERE status = PUBLISHED)
in_progress_count = COUNT(items WHERE status = IN-PROGRESS)
pending_count     = COUNT(items WHERE status = PENDING)
skipped_count     = COUNT(items WHERE status = SKIPPED)
blocked_count     = COUNT(items WHERE status = BLOCKED)
completion_rate   = published_count / (total_items - skipped_count) × 100
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**Phase 4B — Last Published Item Tracking**

After every PUBLISHED status update:
1. Set `last_published_item` to the newly published `item_id`.
2. Set `last_published_date` to the current timestamp.
3. Set `last_progress_date` to the current timestamp.
4. Reset `sessions_without_progress` to 0.

**Phase 4C — Next Pending Item Identification**

After every status update, recalculate `next_pending_item`:
1. Filter all items with status = PENDING.
2. Verify none of those items have unresolved dependencies (items in their `depends_on` list that are still PENDING or IN-PROGRESS).
3. Of the PENDING items with no blocking dependencies: sort by `priority` (HIGH first) then by `month` (lowest first).
4. Set `next_pending_item` = the top item's `item_id`.

**Phase 4D — Metrics History**

Store a snapshot of metrics at the end of every session:

```
METRICS SNAPSHOT (stored per session)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
session_id          : [UUID]
session_date        : [ISO 8601]
published           : [N]
in_progress         : [N]
pending             : [N]
skipped             : [N]
blocked             : [N]
completion_rate     : [N]%
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

This snapshot history enables strategy evolution tracking and velocity analysis.

### ▸ STEP 5 — STRATEGIC DECISION LOGGING

Store every strategic decision produced by `strategist-ai.skill` and confirmed by the user or system.

**Phase 5A — Decision Intake**

Strategic decisions arrive from two sources:
1. `strategist-ai.skill` — Next Best Actions and Avoidance items from its strategic brief.
2. `nexus-connector.skill` — Execution mode decisions, pipeline choices, and configuration decisions.

For each decision, collect:
- The decision statement (must be specific — minimum 10 characters; not a category label).
- The rationale (must be data-backed — references specific skills, scores, or metrics).
- The source skill.
- The current session_id.
- Auto-set `date` = current timestamp.

**Phase 5B — Decision Quality Validation**

Before storing any decision, validate its quality:

| Validation Check                                                         | Pass Condition                                              |
|--------------------------------------------------------------------------|-------------------------------------------------------------|
| Decision statement is specific                                            | ≥ 10 characters; names a specific action or direction      |
| Decision statement is not a category                                     | "Prioritize BOFU content" → reject; "Create BOFU comparison page targeting 'crm software for teams'" → pass |
| Rationale is data-backed                                                 | References at least one specific skill, metric, or cluster |
| Decision is not a duplicate of an existing active decision              | No existing active decision with ≥ 80% semantic similarity |

**Phase 5C — Decision Tagging**

Assign tags to each decision based on its subject matter:

| Tag              | Decision Type                                                              |
|------------------|----------------------------------------------------------------------------|
| CLUSTER          | Decision relates to which clusters to build or prioritize                 |
| KEYWORD          | Decision relates to specific keyword targeting or avoidance               |
| CONTENT          | Decision relates to content format, type, or structure                    |
| FUNNEL           | Decision relates to funnel stage prioritization                           |
| COMPETITIVE      | Decision relates to competitor positioning or differentiation             |
| STRUCTURAL       | Decision relates to site architecture, internal linking, or URL structure |

**Phase 5D — Decision Storage**

Store the validated, tagged decision record:
1. Generate `decision_id` (UUID).
2. Set `active` = true.
3. Write the full record to `strategy:decision:[project_id]:[decision_id]`.
4. Add the `decision_id` to `strategy:decisions:[project_id]` index list.

### ▸ STEP 6 — RECOMMENDATION TRACKING

Log every recommendation from the system and track whether it was acted on and what the outcome was.

**Phase 6A — Recommendation Intake**

Recommendations differ from decisions in their specificity and immediacy:
- A **decision** is a strategic choice that the project commits to.
- A **recommendation** is a suggestion that the system offers; the user may or may not act on it.

Recommendations arrive from:
- `strategist-ai.skill` (Next Best Actions, Avoidance items).
- `optimization-engine.skill` (content improvement suggestions).
- `gap-opportunity-engine.skill` (content gap suggestions).

**Phase 6B — Recommendation Storage**

For each recommendation:
1. Generate `recommendation_id` (UUID).
2. Set `acted_upon` = false.
3. Set `dismissed` = false.
4. Store the full recommendation text, source skill, session_id, and timestamp.
5. Write to `strategy:recommendation:[project_id]:[recommendation_id]`.

**Phase 6C — Act-Upon Confirmation Flow**

At the start of every session, surface unacted-upon recommendations from the prior session:

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**Phase 6D — Act-Upon Recording**

**Phase 6E — Dismissal Recording**

When a user dismisses a recommendation:

3. Set `dismissed_reason` = user-provided reason.
4. Log: `"RECOMMENDATION DISMISSED: [recommendation_id] — Reason: [reason]"`

### ▸ STEP 7 — CONFLICT DETECTION

Before storing a new strategic decision, check whether it contradicts a prior active decision.

**Phase 7A — Conflict Detection Method**

1. Load all active decisions for this project where `active` = true.
2. For each active decision, compare its semantic meaning against the incoming decision.
3. A conflict exists when:
   - The incoming decision reverses the direction of a prior decision (e.g., prior: "Focus on CRM cluster first"; new: "Focus on Email cluster first — defer CRM").
   - The incoming decision targets the same entity (cluster, keyword, content piece) with opposite instructions.
   - The incoming decision contradicts a data claim that a prior decision was built on.

**Phase 7B — Conflict Classification**

| Conflict Type                                         | Definition                                                         |
|-------------------------------------------------------|--------------------------------------------------------------------|
| **DIRECT CONTRADICTION**                             | Same entity, opposite instruction — cannot both be true           |
| **PRIORITY REVERSAL**                                | Prior decision prioritizes A; new decision de-prioritizes A       |
| **SCOPE CONFLICT**                                   | Prior decision narrows focus; new decision broadens it            |
| **DATA CONFLICT**                                    | New decision is based on a metric that contradicts the prior decision's basis |

**Phase 7C — Conflict Resolution Flow**

When a conflict is detected:

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
⚠️ STRATEGIC CONFLICT DETECTED
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
New Decision    : "[incoming decision text]"
Conflicts With  : "[prior decision text]" (made [date], session [N])

Conflict Type   : [DIRECT CONTRADICTION / PRIORITY REVERSAL / SCOPE CONFLICT]
Conflict Detail : [specific description of how they conflict]

How would you like to resolve this?
[A] Accept new decision — supersede the prior one
[K] Keep prior decision — discard the new one
[B] Keep both — they apply to different contexts (specify)

Enter resolution:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**Phase 7D — Resolution Recording**

| Resolution | Action                                                                                          |
|------------|-------------------------------------------------------------------------------------------------|
| A          | Set prior decision `active` = false; set prior decision `superseded_by` = new decision_id; store new decision with `conflicts_with` = [prior decision_id] + `conflict_resolution` = "SUPERSEDED" |
| K          | Discard new decision; log: "New decision discarded — prior decision retained." |
| B          | Store both as active; set `conflicts_with` on both; set `conflict_resolution` = user-provided context text |

### ▸ EXECUTION GAP DETECTION

Identify roadmap items that are stuck in IN-PROGRESS without progression.

**Execution Gap Detection Protocol:**

1. Query all items with status = IN-PROGRESS.
2. For each IN-PROGRESS item, calculate days since `status_updated_date`.
3. Apply gap thresholds:

| Days IN-PROGRESS without Published | Classification    | Action                                              |
|-------------------------------------|-------------------|-----------------------------------------------------|
| 7–14 days                           | NORMAL            | No action                                           |
| 15–21 days                          | AT RISK           | Flag in progress summary as "Taking longer than expected" |
| 22–30 days                          | EXECUTION GAP     | Surface as alert at session start                   |
| > 30 days                           | STALLED           | Escalate — prompt user to update status or explain |

**Execution Gap Alert:**

```
> **⚠️ EXECUTION GAP:** "[topic]" has been IN-PROGRESS for [N] days with no published status.
> Either update its status to PUBLISHED (with URL) or SKIPPED (with reason).
> Unresolved execution gaps distort your completion rate and block dependent items.
```

### ▸ STAGNATION DETECTION

Detect when no execution progress has been made across multiple sessions.

**Stagnation Detection Protocol:**

1. At every session start, check `sessions_without_progress`.
2. If `last_progress_date` did not change since the last session → increment `sessions_without_progress`.
3. If `sessions_without_progress` ≥ 3 → set `stagnation_flag` = true.

**Stagnation Alert (when stagnation_flag = true):**


## SECTION 7 — DEDUPLICATION RULES
## ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

### Rule 1 — No Duplicate Roadmap Items Within the Same Version

Within a single roadmap version, no two items may have the same `target_keyword`. A roadmap cannot include the same keyword twice — each keyword maps to exactly one content piece. If a duplicate keyword is detected during roadmap storage → reject the duplicate item; log: `"Duplicate keyword '[keyword]' in roadmap v[N] — second instance rejected. Only one item per target keyword."`

### Rule 2 — No Repeated Strategic Decisions

No two active strategic decisions for the same project may have ≥ 80% semantic similarity in their `decision` text. When a near-duplicate is detected → trigger Step 7 conflict detection. Outcomes: the newer decision supersedes the older one (A), or the newer one is discarded (K).

### Rule 3 — No Duplicate Keyword Assignments Across Active and Planned Content

A roadmap item's `target_keyword` must not duplicate a keyword that already has a PUBLISHED or DRAFT content record in `content-memory.skill`. Before storing a roadmap item, the keyword is cross-checked against `content-memory.skill`. If a PUBLISHED record exists for the keyword → the roadmap item should instead be a REFRESH item (item_type = REFRESH), not a NEW item.

### Rule 4 — No Duplicate Recommendation Records

No two recommendation records may have the same recommendation text AND the same source_skill within the same session. Duplicate recommendations produced in the same session (identical text from the same skill) are consolidated into one record.

### Rule 5 — One Active Roadmap Per Project

Each project may have exactly one roadmap with status = ACTIVE at any time. Attempting to activate a second roadmap while one is ACTIVE triggers the prior roadmap handling flow (Phase 1C) — never silent overwrite.

## SECTION 8 — ANTI-FLUFF RULES
## ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

### Prohibited Strategy Memory Behaviors

| Prohibited Behavior                                                                              | Reason                                                                            |
|--------------------------------------------------------------------------------------------------|-----------------------------------------------------------------------------------|
| Storing a strategic decision without a rationale                                                  | Decisions without rationale are undebatable — the reason behind a decision is as important as the decision itself |
| Storing a vague decision like "Prioritize SEO" or "Create more content"                          | Decisions must be specific and actionable — vague decisions cannot be tracked, executed, or reversed |
| Marking a recommendation as acted_upon without asking for the outcome                            | An acted-upon recommendation without an outcome is an incomplete tracking record  |
| Storing a roadmap without confirming it with the user first                                      | Unconfirmed roadmaps must be in DRAFT status — activation requires explicit user consent |
| Allowing the roadmap to show PENDING items where the dependency (blocking item) has already been published but the BLOCKED item was not unblocked | Dependency resolution must be automatic — stale BLOCKED status is a data integrity failure |
| Tracking sessions_without_progress without triggering the stagnation alert                      | The counter exists to trigger the alert — a counter without an alert is unused data |
| Closing a session without updating `last_progress_date` (even if it was unchanged)              | The date comparison requires every session end to be recorded                    |
| Storing a SKIPPED item without a skipped_reason                                                  | Skipped items with no reason become invisible black holes in the roadmap history  |
| Auto-resolving a strategy conflict without user input                                            | Conflicts are resolved by the user — the system presents options, never makes the choice |
| Allowing duplicate active roadmaps to coexist without forcing resolution                        | Dual active roadmaps produce an undefined "next action" state                    |

### Required Strategy Memory Behaviors

- Every roadmap item must have: topic, target_keyword, intent, cluster, priority, phase, month, rationale, and status.
- Every strategic decision must have: decision text, rationale, source_skill, session_id, date.
- Every recommendation must have: recommendation text, source_skill, session_id, date, acted_upon status.
- Every session start triggers the continuity context package and progress summary.
- Stagnation check runs at every session start — no exceptions.
- Execution gap check runs at every session start — surfaces all IN-PROGRESS items older than 21 days.

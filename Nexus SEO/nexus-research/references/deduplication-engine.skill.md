## SECTION 1 — ROLE
## ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

This skill is the **central quality control system** of the Nexus SEO Operating System.

Every keyword list, content idea set, cluster assignment, and content plan generated by any Nexus skill must pass through this engine before it is delivered to the user, written to memory, or forwarded to a downstream skill. The deduplication engine is the single enforcement point for the system's commitment that no redundant, overlapping, or conflicting output will ever reach execution.

Duplication in SEO is not a cosmetic problem — it is a performance problem. Duplicate keywords create internal competition. Duplicate content pieces cause cannibalization. Overlapping intent causes SERP confusion. Conflicting pipeline items waste editorial resources. This skill prevents all of these outcomes systematically by running a multi-level, eight-step deduplication protocol across every form of redundancy the system can produce.

### Primary Responsibilities

| Responsibility                          | Description                                                                                                         |
|-----------------------------------------|---------------------------------------------------------------------------------------------------------------------|
| **Exact Deduplication**                 | Remove identical keywords and content ideas — including case-insensitive and whitespace-normalized duplicates        |
| **Semantic Deduplication**              | Detect and remove keywords and ideas that convey the same meaning through different phrasing                        |
| **Intent Deduplication**                | Identify keywords that serve the same user goal at the same funnel stage and resolve the redundancy                  |
| **Cluster Deduplication**               | Prevent the same keyword from appearing in multiple topic clusters; apply reference tagging instead                 |
| **Pipeline Awareness Checking**         | Cross-reference all candidates against published, planned, and draft content to block redundant production           |
| **Refresh Exception Handling**          | Identify candidates that match existing content but merit a refresh rather than rejection                           |
| **Cross-Session Deduplication**         | Check memory for keywords and topics used in prior sessions; block repetition across session boundaries             |
| **Clean Output Production**             | Deliver a fully deduplicated keyword or content list to downstream skills                                           |
| **Rejection Log Production**            | Produce a complete, justified rejection log for every item removed — no blind removals, no silent drops             |
| **Cannibalization Detection**           | Identify when new candidates would compete with existing published or planned content for the same SERP position     |

### Mandatory Invocation Rule

This skill is NOT optional. It MUST be called by:

- `keyword-discovery.skill` — before outputting any keyword list.
- `semantic-clustering.skill` — before finalizing any cluster keyword assignment.
- `serp-blueprint-generator.skill` — before finalizing keyword expansion sections.
- `content-generation-engine.skill` — before accepting a content configuration.
- `roadmap-engine.skill` — before generating any content roadmap entry.
- `gap-opportunity-engine.skill` — before finalizing any opportunity recommendation.
- Any other skill that produces keywords, content ideas, or content plans as output.

**No skill in the Nexus system may produce keyword or content output without calling this engine first.**

### What This Skill Is NOT

- It is **not** a keyword ranker or scorer. It determines what is redundant — not what is most valuable.
- It is **not** a keyword generator. It cleans what was generated — it does not create new alternatives.
- It is **not** a content quality evaluator. It prevents duplication — it does not evaluate writing quality or depth.
- It is **not** a decision-maker for borderline cases. When uncertain, it keeps the keyword and flags it for human review. It does not remove items it is not certain about.
- It is **not** a replacement for gap analysis. It prevents redundancy — `gap-opportunity-engine.skill` identifies opportunities.

### The Defining Standard of This Skill

> Every rejection produced by this skill must have a specific, evidence-backed reason.
> No keyword or content idea may be silently dropped.
> The rejection log is as important as the clean list — it is the audit trail of the system's quality control.
>
> When uncertain: KEEP the item. Flag it. Surface it for human review.
> It is better to keep a borderline duplicate than to silently remove a valid strategic keyword.

### ▸ STEP 1 — EXACT DEDUPLICATION

The first and simplest level of deduplication. Remove candidates that are identical to another candidate in the same input batch or to an item already in the reference databases.

**Phase 1A — Pre-Processing Normalization**

Before any comparison, normalize every candidate through the same transformation pipeline:

```
NORMALIZATION PIPELINE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Input: Raw candidate string

Step 1: Lowercase all characters
  "Best CRM Software" → "best crm software"

Step 2: Trim leading and trailing whitespace
  " best crm software " → "best crm software"

Step 3: Collapse internal whitespace
  "best  crm   software" → "best crm software"

Step 4: Normalize punctuation
  "best crm software!" → "best crm software"
  "best crm software?" → "best crm software" (preserve for question-type detection in Step 2)
  Exception: preserve hyphens within compound terms ("e-commerce", "how-to")

Step 5: Expand common abbreviations
  "seo" → "seo" (keep — domain standard)
  "crm" → "crm" (keep — domain standard)
  "b2b" → "b2b" (keep — domain standard)
  Note: Do not expand if the abbreviation IS the keyword (search users use abbreviations)

Step 6: Strip duplicate spaces from step 4/5
  Final normalized form: clean lowercase string

Output: normalized_candidate (string)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**Phase 1B — Within-Batch Exact Match Detection**

Compare every candidate's normalized form against every other candidate's normalized form in the same input batch:

1. Build a hash map of normalized forms → original candidates.
2. For each normalized form: if it already exists in the hash map → the new candidate is an EXACT DUPLICATE.
3. Keep the first instance. Reject all subsequent instances.
4. Log each rejection: `"EXACT DUPLICATE of [kept candidate] — same string after normalization."`

**Phase 1C — Cross-Reference Exact Match Detection**

Compare every surviving candidate against the three reference databases:

| Reference Check                                | Match Condition                                              | Action                              |
|------------------------------------------------|--------------------------------------------------------------|-------------------------------------|
| vs. Published Content DB (primary keywords)    | Normalized candidate = normalized primary keyword            | REJECT — PUBLISHED EXACT MATCH      |
| vs. Published Content DB (secondary keywords)  | Normalized candidate = normalized secondary keyword          | FLAG — SECONDARY KEYWORD OVERLAP (lower severity; check Step 2) |
| vs. Pipeline Content DB (target keywords)      | Normalized candidate = normalized pipeline target keyword    | REJECT — PIPELINE EXACT MATCH       |
| vs. Memory DB (prior session keywords)         | Normalized candidate = normalized memory keyword             | REJECT — MEMORY EXACT MATCH         |

**Phase 1D — URL Deduplication (for content ideas)**

When candidates are content ideas (working titles) rather than keywords:

1. Generate a slug-equivalent from the working title.
2. Compare against all URLs in the Published Content DB.
3. If a URL slug matches the working title slug → REJECT — URL DUPLICATE.

### ▸ STEP 2 — SEMANTIC DEDUPLICATION

Beyond exact matches, detect candidates that express the same meaning through different phrasing. Semantic duplicates share the same core topic, the same verb-object relationship, and the same informational value — they would produce identical or near-identical content if executed.

**Phase 2A — Semantic Equivalence Test**

For each surviving candidate, test it against all other surviving candidates AND against the reference databases for semantic equivalence.

**The Five Semantic Equivalence Signals:**

A pair of keywords is semantically equivalent if it satisfies ≥3 of the following 5 conditions:

| Signal                            | Condition                                                                        |
|-----------------------------------|----------------------------------------------------------------------------------|
| **Same core verb + same object**  | Both keywords contain the same action and the same target concept               |
| **Same question intent**          | Both keywords, when framed as a question, ask the same thing                    |
| **Phrasing variation only**       | The difference between the two is word order, synonym substitution, or formality level |
| **Same featured snippet target**  | Both keywords would naturally produce the same content as a featured snippet answer |
| **Same SERP overlap**             | Both keywords would target the same top-3 SERP results if searched today        |

**Semantic Equivalence Examples:**

| Keyword A                              | Keyword B                              | Signals Met    | Verdict                |
|----------------------------------------|----------------------------------------|----------------|------------------------|
| `"how to do an seo audit"`             | `"seo audit guide"`                    | 5/5            | SEMANTIC DUPLICATE     |
| `"best crm for startups"`              | `"top crm tools for startups"`         | 4/5            | SEMANTIC DUPLICATE     |
| `"crm software for small business"`    | `"small business crm"`                 | 4/5            | SEMANTIC DUPLICATE     |
| `"email marketing best practices"`     | `"email marketing tips"`               | 3/5            | SEMANTIC DUPLICATE     |
| `"hubspot vs salesforce"`              | `"hubspot vs salesforce 2024"`         | 5/5            | SEMANTIC DUPLICATE — year variant |
| `"how to write a blog post"`           | `"blog post writing guide"`            | 4/5            | SEMANTIC DUPLICATE     |
| `"crm for real estate"`                | `"crm for healthcare"`                 | 1/5            | NOT a duplicate — different audiences |
| `"best crm software"`                  | `"best project management software"`   | 0/5            | NOT a duplicate — different categories |

**Phase 2B — Synonym and Reformulation Detection**

Beyond the 5-signal test, detect these specific semantic reformulation patterns:

| Pattern Type                              | Example Pair                                               | Detection Rule                                             |
|-------------------------------------------|------------------------------------------------------------|------------------------------------------------------------|
| **Synonym substitution**                  | `"website traffic"` vs `"web traffic"`                    | One word substituted with a recognized synonym            |
| **Word order reversal**                   | `"crm software best"` vs `"best crm software"`           | Same words, different order — normalize word order for comparison |
| **Question vs. statement**                | `"what is seo"` vs `"seo explained"`                     | One is a question; the other is the answer statement form |
| **Plural/singular variation**             | `"seo tool"` vs `"seo tools"`                            | Difference is only plural vs. singular form               |
| **With/without qualifier**                | `"crm software"` vs `"crm software review"`              | One has a trailing format qualifier the other lacks       |
| **Contraction/expansion**                 | `"email open rate"` vs `"email opening rate"`             | One form is contracted; the other expanded                |
| **Year variant**                          | `"best seo tools 2023"` vs `"best seo tools 2024"`       | Same query, different year — same SERP target             |

**Phase 2C — Semantic Deduplication Resolution**

When a semantic duplicate pair is detected:

**Keep/Remove Decision Logic:**

| Condition                                                          | Keep                                  | Remove                               |
|--------------------------------------------------------------------|---------------------------------------|--------------------------------------|
| One is more specific (has an audience modifier)                    | The more specific one                 | The generic one                      |
| One is a question form; the other is a statement form              | The question form (higher AEO value) | The statement form                   |
| One has a year variant; the other does not                         | The current-year variant              | The non-year variant                 |
| One has a higher search specificity (more words, more specific)    | The more specific one                 | The head term                        |
| Both are equally specific — no meaningful differentiator           | Keep the one discovered first         | Remove the second                    |
| One is in the reference database (published/pipeline/memory)       | The one in the database — note as PUBLISHED/PLANNED | The candidate — REJECT as semantic dup |

Log every semantic duplicate rejection with: `"SEMANTIC DUPLICATE — [reason]: [kept keyword]. Keywords share [N/5 signals]."`

### ▸ STEP 3 — INTENT DEDUPLICATION

Two keywords can be semantically distinct (covering different facets of a topic) but still serve the same user goal at the same funnel stage. Intent duplicates produce content that would compete for the same reader at the same decision point — even if the articles themselves would be written differently.

**Phase 3A — Intent Extraction**

For every surviving candidate, extract or confirm its intent classification:

| Intent Classification Source                        | Priority |
|-----------------------------------------------------|----------|
| Intent field provided in the input packet           | 1 (use directly) |
| Intent inferred from the invocation context         | 2 |
| Intent inferred from keyword pattern (Step 4 of content-inventory.skill logic) | 3 |
| Intent unknown → classify using URL pattern rules   | 4 |

**Phase 3B — Intent Duplicate Detection**

Two candidates are intent duplicates when:

1. They share the same primary intent classification (TOFU/MOFU/BOFU).
2. They address the same core user question or decision.
3. A reader who searched either keyword would have the same primary informational need.

**Intent Duplicate Detection Matrix:**

| Candidate A                                | Candidate B                                | Same Intent? | Same User Need? | Intent Duplicate? |
|--------------------------------------------|--------------------------------------------|--------------|-----------------|--------------------|
| `"best crm software"` (MOFU)              | `"top crm tools"` (MOFU)                  | YES          | YES             | YES                |
| `"best crm software"` (MOFU)              | `"crm software comparison"` (MOFU)         | YES          | YES             | YES                |
| `"best crm for startups"` (MOFU)          | `"best crm for enterprise"` (MOFU)         | YES          | NO (different audience) | NO — distinct audiences |
| `"how to set up crm"` (TOFU)              | `"crm implementation guide"` (TOFU)        | YES          | YES             | YES                |
| `"what is crm"` (TOFU)                    | `"crm pricing"` (BOFU)                    | NO           | NO              | NO — different funnel stages |
| `"hubspot vs salesforce"` (MOFU)          | `"hubspot alternatives"` (MOFU)            | YES          | NO (vs vs alternatives) | NO — distinct comparison types |

**Phase 3C — Intent Duplicate Resolution**

When an intent duplicate pair is detected:

**Keep/Remove Decision Logic:**

| Condition                                                         | Keep                                | Remove                             |
|-------------------------------------------------------------------|-------------------------------------|------------------------------------|
| One is clearer — more precisely expresses the user's goal        | The clearer keyword                 | The vague variation               |
| One is more specific (has audience, use case, or tool modifier)   | The more specific keyword           | The generic version               |
| One is in the reference databases                                 | The reference database entry        | The new candidate (REJECT)        |
| Neither is clearly superior                                       | Keep both; flag both as REVIEW      | Neither removed                   |
| Both are new candidates; one is more searchable                   | The more natural, searchable phrasing | The awkward or reversed phrasing |

Log every intent duplicate rejection with: `"INTENT DUPLICATE — same user goal ([goal description]) at [funnel stage] as [kept keyword]."`

### ▸ STEP 4 — CLUSTER DEDUPLICATION

Within the cluster architecture, no keyword may appear in more than one cluster. Cluster deduplication enforces this rule — and when a keyword belongs to multiple clusters conceptually, it applies reference tagging rather than duplication.

**Phase 4A — Cluster Membership Check**

For every surviving candidate, determine its cluster assignment:

1. **From input:** If the candidate arrives with a cluster assignment from `semantic-clustering.skill` → use that assignment.
2. **From keyword memory:** If the candidate's normalized form matches a keyword already in `keyword-memory.skill` → retrieve its cluster assignment.
3. **From content inventory:** If the candidate matches a published page's keyword → retrieve the cluster from the content inventory.
4. **Inferred:** If no assignment is available → infer from keyword context.

**Phase 4B — Cross-Cluster Conflict Detection**

A cross-cluster conflict exists when:
- A candidate is being assigned to Cluster A.
- The same keyword (exact or semantic duplicate from Steps 1–3) already exists in Cluster B.
- Cluster A and Cluster B are different clusters.

**Cross-Cluster Resolution Protocol:**

| Conflict Type                                                        | Resolution                                                                          |
|----------------------------------------------------------------------|-------------------------------------------------------------------------------------|
| Exact keyword in Cluster A; same keyword being added to Cluster B   | REJECT from Cluster B. Log: "CLUSTER DUPLICATE — exists in [Cluster A]."           |
| Keyword is genuinely a hub topic shared across two clusters          | Keep in the primary cluster; add CROSS-REFERENCE TAG to the other cluster          |
| Keyword is semantically appropriate for two clusters but not exact   | Keep in the more thematically aligned cluster; add SECONDARY MENTION note to the other |
| Keyword was previously assigned to a cluster now being restructured  | Flag as CLUSTER REASSIGNMENT NEEDED — do not auto-assign to new cluster            |

**Phase 4C — Reference Tagging (Alternative to Duplication)**

When a keyword legitimately connects two clusters (e.g., a keyword that bridges a foundational concept and an advanced application):

Instead of duplicating the keyword across clusters, apply a cross-reference tag:

```
Cluster A: [Cluster Name]
  Keyword: [keyword] — PRIMARY ASSIGNMENT
  Cross-reference: [Cluster B Name] (keyword has secondary relevance here)

Cluster B: [Cluster Name]
  Keyword: [keyword] — CROSS-REFERENCED from [Cluster A]
  Note: Content for this keyword lives in [Cluster A]; internal link recommended from [Cluster B]
```

This reference tag is written to `keyword-memory.skill` via `memory-controller.skill` and used by `internal-linking.skill` to construct appropriate cross-cluster links.

### ▸ STEP 5 — PIPELINE AWARENESS CHECKING

Cross-reference every surviving candidate against the complete content pipeline — published, planned, and draft — to block redundant content production.

**Phase 5A — Published Content Cross-Reference**

For every surviving candidate:

1. Search the Published Content DB for:
   - Primary keyword match (exact or semantic — applying rules from Steps 1 and 2).
   - Secondary keyword match — candidate is a secondary keyword for an existing page.
   - Topic equivalence — candidate covers the same topic as an existing page from the same angle.

2. If a match is found:
   - **Standard rejection:** Mark as REJECT — PUBLISHED. Log: `"Already covered at [URL] — published [date] — coverage score [N]."`
   - **Exception for refresh:** Proceed to Step 6 to determine if a refresh is warranted instead.

**Phase 5B — Draft and Planned Content Cross-Reference**

For every surviving candidate:

1. Search the Pipeline Content DB for:
   - Target keyword match (exact or semantic).
   - Working title match — candidate's topic matches a planned piece's working title.

2. If a match is found:
   - Mark as REJECT — PIPELINE CONFLICT.
   - Log: `"Already in pipeline as [working title] — status: [draft/planned] — keyword: [target keyword]."`

**Phase 5C — Cannibalization Risk Assessment**

Beyond direct matches, detect candidates that would create cannibalization risk against existing or planned content without being exact matches:

**Cannibalization Test:**

"If both the candidate content and the existing/planned content were published and fully optimized, would they compete for the same SERP position?"

| Cannibalization Condition                                                       | Classification                     |
|---------------------------------------------------------------------------------|------------------------------------|
| Same cluster + same intent + same target audience + semantically adjacent keywords | HIGH cannibalization risk       |
| Same cluster + same intent + different audience                                  | LOW cannibalization risk — different audience creates natural differentiation |
| Same cluster + different intent (e.g., TOFU vs. BOFU)                           | NO cannibalization — different funnel stages do not compete |
| Different clusters + same keyword                                               | Cluster assignment error — resolve via Step 4, not here |

**Cannibalization Resolution:**

| Risk Level | Action                                                                             |
|------------|------------------------------------------------------------------------------------|
| HIGH       | REJECT candidate — flag as CANNIBALIZATION RISK with reference to existing content|
| LOW        | KEEP candidate — flag as CANNIBALIZATION WATCH for monitoring                     |
| NONE       | No action — proceed normally                                                       |

Log each high-risk cannibalization: `"CANNIBALIZATION RISK — candidate would compete with [URL/title] for [SERP position description] at [intent] in [cluster]."`

### ▸ STEP 6 — REFRESH EXCEPTION HANDLING

Step 5 would normally reject any candidate that matches existing published content. This step intercepts those rejections and evaluates whether the matching published content is outdated, thin, or underperforming — conditions that warrant an UPDATE rather than a rejection.

**Phase 6A — Refresh Qualification Criteria**

An item is a REFRESH CANDIDATE (instead of a simple rejection) if ALL of the following conditions are met:

| Condition                                           | Qualifying Threshold                                                   |
|-----------------------------------------------------|------------------------------------------------------------------------|
| **Matching existing published content**             | The candidate matches an existing URL's keyword or topic               |
| **Content age**                                     | The matching content is STALE (12–24 months) or OUTDATED (24+ months) |
| **OR thin coverage**                                | The matching content has a coverage score of 1 or 2                   |
| **OR SERP underperformance**                        | The matching content is outranked by newer, fresher competitor content |
| **Candidate keyword is not itself stale**           | The candidate keyword is still relevant and actively searched          |

**Refresh Qualification Matrix:**

| Published Content Freshness | Coverage Score | SERP Performance | Refresh Decision        |
|-----------------------------|----------------|------------------|-------------------------|
| FRESH (< 6 months)          | 4–5            | Good             | REJECT — no refresh needed |
| FRESH (< 6 months)          | 1–2            | Any              | REFRESH — thin content regardless of age |
| AGING (6–12 months)         | 3–5            | Good             | REJECT — still current  |
| AGING (6–12 months)         | 1–2            | Poor             | REFRESH — thin + aging  |
| STALE (12–24 months)        | Any            | Any              | REFRESH — update recommended |
| OUTDATED (24+ months)       | Any            | Any              | REFRESH — urgent update |
| CRITICALLY OUTDATED (36+mo) | Any            | Any              | REFRESH — critical update |

**Phase 6B — Refresh Candidate Output**

When an item qualifies as a REFRESH CANDIDATE:

1. Do NOT reject it from the pipeline.
2. Do NOT add it to the clean list as a new candidate.
3. Add it to the **Refresh Candidates** section of the clean list output.
4. Log in the rejection log with type: REFRESH CANDIDATE (not REJECTED).

**Refresh Candidate Record Format:**

```
REFRESH CANDIDATE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Candidate Keyword   : [keyword]
Matching URL        : [URL]
Match Type          : [EXACT / SEMANTIC / TOPIC]
Content Age         : [N months]
Coverage Score      : [N/5]
Refresh Trigger     : [AGE / THIN CONTENT / SERP UNDERPERFORMANCE / COMBINED]
Refresh Priority    : [CRITICAL / HIGH / MEDIUM / LOW]
Recommended Action  : [Update existing page / Major rewrite / Section addition / Data refresh]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**Phase 6C — Refresh vs. New Content Decision**

In some cases, the REFRESH CANDIDATE is so outdated that a fresh piece would be more effective than updating the original. Apply the following decision logic:

| Condition                                                               | Recommendation                                    |
|-------------------------------------------------------------------------|---------------------------------------------------|
| Content is OUTDATED/CRITICALLY OUTDATED AND coverage score 1–2         | NEW CONTENT — retire old; redirect to new         |
| Content is STALE AND coverage score 3+                                  | REFRESH — update existing                         |
| Content is STALE AND coverage score 1–2                                 | REFRESH with major rewrite — or new content       |
| Content is AGING AND coverage score 4–5                                 | DATA REFRESH only — update statistics and dates   |
| URL structure is outdated (year in URL, obsolete format)                | NEW CONTENT — old URL is a liability              |

### ▸ STEP 7 — CROSS-SESSION DEDUPLICATION

Check the memory database for keywords and topics that were generated, used, or rejected in prior sessions. This prevents the system from cycling through the same keyword territory indefinitely across multiple work sessions.

**Phase 7A — Memory Database Query**

Query `memory-controller.skill` for:

1. All keywords that were ADDED to the keyword universe in prior sessions for this domain.
2. All keywords that were REJECTED in prior sessions and the reason for rejection.
3. All content topics that were ASSIGNED to planned content in prior sessions.
4. All content topics that were PUBLISHED in prior sessions (and their current lifecycle state).
5. All keywords that were ABANDONED — researched in prior sessions but never assigned.

**Phase 7B — Memory Duplicate Detection**

For every surviving candidate:

1. Check if the normalized candidate appears in any of the five memory categories above.
2. Apply category-specific rules:

| Memory Category                           | Detection Match                    | Action                                                        |
|-------------------------------------------|------------------------------------|---------------------------------------------------------------|
| Previously ADDED to keyword universe      | Exact or semantic match            | REJECT — MEMORY DUPLICATE. Log: `"Generated in session [ID] on [date]."`|
| Previously REJECTED (same reason valid)   | Exact match + rejection reason still valid | REJECT — MEMORY DUPLICATE (re-rejection). Log reason. |
| Previously REJECTED (reason no longer valid) | Exact match + rejection was for freshness/SERP (now changed) | KEEP — MEMORY EXCEPTION. Note the prior rejection and why it no longer applies. |
| Previously ASSIGNED (content planned)     | Exact or semantic match            | REJECT — PIPELINE CONFLICT (cross-session). Log: `"Assigned in session [ID]."`|
| Previously PUBLISHED                      | Exact or semantic match            | Defer to Step 5/6 logic — published cross-reference           |
| Previously ABANDONED                      | Exact or semantic match            | KEEP — with flag: "Previously abandoned in session [ID] — verify if still relevant." |

**Phase 7C — Memory Exception Protocol**

When a candidate was previously rejected in memory but the rejection reason is no longer valid:

| Prior Rejection Reason                                   | Validity Now                              | Action                                      |
|----------------------------------------------------------|-------------------------------------------|---------------------------------------------|
| "PUBLISHED — thin content at [URL]"                      | URL no longer exists / was deleted        | KEEP — publication reference is void        |
| "PIPELINE CONFLICT — planned as [title]"                 | Planned piece was cancelled               | KEEP — pipeline reference is void           |
| "STALE SERP — not worth targeting"                       | SERP has changed; topic is now relevant   | KEEP — SERP context has evolved             |
| "SEMANTIC DUPLICATE of [keyword]"                        | The referenced keyword was removed        | KEEP — duplicate reference no longer exists |
| "OUT OF SCOPE for this domain"                           | Domain scope has expanded                 | KEEP — with SCOPE EXPANSION flag            |

All memory exceptions are logged: `"MEMORY EXCEPTION — prior rejection on [date] in session [ID] is no longer valid because [reason]."`

### ▸ STEP 8 — FINAL OUTPUT PRODUCTION

After all 7 deduplication steps are complete, produce the clean list and rejection log.

**Phase 8A — Final Status Assignment**

For every candidate that survived all steps, assign a final status:

| Status        | Condition                                                                                 |
|---------------|-------------------------------------------------------------------------------------------|
| **CLEAN**     | Passed all steps without any flags; high confidence it is unique and valid               |
| **REVIEW**    | Passed all steps but has a MEDIUM-confidence flag (borderline semantic match, abandoned memory entry, scope expansion exception) |
| **REFRESH**   | Matches existing content but qualifies for refresh treatment (Step 6)                    |

**Phase 8B — Confidence Assignment**

For every clean or review item, assign a deduplication confidence level:

| Confidence | Condition                                                                       |
|------------|---------------------------------------------------------------------------------|
| HIGH       | All steps ran at full depth; no borderline flags; clear unique status           |
| MEDIUM     | At least one borderline check was encountered; a human should verify this item  |
| LOW        | Multiple borderline flags; uncertain about uniqueness; kept by failsafe rule    |

**Phase 8C — Rejection Log Finalization**

After all rejections are recorded:

1. Sort the rejection log by step (Step 1 first, memory rejections last).
2. Count rejections by type.
3. Produce the rejection summary by step.
4. Flag the rejection log as AUDIT TRAIL — it must be preserved in `memory-controller.skill`.

**Phase 8D — Output Dispatch**

After both outputs are complete:

1. Return the **Clean List** to the calling skill for downstream processing.
2. Store the **Rejection Log** in `memory-controller.skill` under the current session ID.
3. Update `keyword-memory.skill` via `memory-controller.skill`:
   - Mark all CLEAN items as DEDUP_PASSED with current timestamp.
   - Mark all REJECTED items as DEDUP_REJECTED with reason and timestamp.
   - Mark all REFRESH items as REFRESH_FLAGGED with priority and timestamp.
4. Notify the calling skill of the counts: `[N] passed, [N] rejected, [N] refresh candidates, [N] for review.`

## SECTION 5 — ADVANCED LOGIC
## ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

### ▸ SEMANTIC SIMILARITY DETECTION — ADVANCED IMPLEMENTATION

Beyond the 5-signal test in Step 2, this advanced implementation detects semantic similarity at the concept level — not just the keyword level.

**Concept Extraction and Comparison:**

For every candidate keyword, extract:
1. **Core concept:** The primary noun phrase (what the keyword is about).
2. **Action frame:** The verb or implied action (how, what, why, best, compare).
3. **Modifier frame:** Any audience, use case, or constraint modifier (for startups, without coding, under $50).

Compare concept triplets across all candidates:

| Candidate A Triplet                            | Candidate B Triplet                            | Match Level        |
|------------------------------------------------|------------------------------------------------|--------------------|
| Concept: CRM | Action: best | Modifier: startups | Concept: CRM | Action: top | Modifier: startups | SEMANTIC DUPLICATE |
| Concept: CRM | Action: how-to-set-up | Modifier: — | Concept: CRM | Action: implementation | Modifier: — | SEMANTIC DUPLICATE |
| Concept: CRM | Action: best | Modifier: real-estate | Concept: CRM | Action: best | Modifier: healthcare | NOT DUPLICATE — different modifiers create distinct content |
| Concept: CRM | Action: what-is | Modifier: — | Concept: CRM | Action: pricing | Modifier: — | NOT DUPLICATE — different action frames |

**Semantic Similarity Scoring:**

| Components Matching | Similarity Score | Verdict      |
|---------------------|------------------|--------------|
| All 3 match         | 1.0 (100%)       | SEMANTIC DUPLICATE |
| Core + Action match; Modifier differs | 0.75 (75%) | SEMANTIC DUPLICATE — same topic, same approach, different angle |
| Core matches; Action and Modifier differ | 0.4 (40%) | NOT DUPLICATE — distinct keyword purpose |
| No components match | 0.0 (0%)         | NOT DUPLICATE |

Threshold for semantic duplication: ≥75% similarity score.

---

### ▸ INTENT CLUSTERING COMPARISON — ADVANCED IMPLEMENTATION

When processing a large keyword set (>50 candidates), apply intent clustering to detect intent duplicate groups — multiple candidates that all serve the same user goal — before running individual pair comparisons.

**Intent Cluster Formation:**

1. Group all candidates by their extracted intent (TOFU/MOFU/BOFU).
2. Within each intent group, sub-group by core concept.
3. Within each core concept sub-group, apply semantic similarity scoring.
4. Any sub-group containing ≥2 candidates with similarity score ≥75% is an INTENT CLUSTER — only the best representative from the cluster passes.

**Best Representative Selection Within Intent Cluster:**

| Selection Criterion                                          | Priority |
|--------------------------------------------------------------|----------|
| Most specific keyword (most words, most modifiers)           | 1        |
| Question form (starts with "how", "what", "why", "when")    | 2 (for TOFU clusters) |
| Most recently validated against SERP context                 | 3        |
| Alphabetically first (tie-breaker only)                      | 4        |

**Intent Cluster Output:**

Instead of rejecting each losing candidate individually, report the intent cluster:

```
INTENT CLUSTER DETECTED
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Cluster: MOFU — Best CRM Comparison Keywords
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Candidates in cluster:
  → "best crm software" — KEPT (most searchable form)
  → "top crm tools" — REJECTED (synonym of kept)
  → "best crm platforms" — REJECTED (synonym of kept)
  → "crm software comparison" — REVIEW (adjacent but distinct enough; check manually)
Kept: 1 | Rejected: 2 | Review: 1
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### ▸ CONTENT-LEVEL DUPLICATION DETECTION

When the input contains content ideas (titles, working topics) rather than raw keywords, apply a deeper form of duplication detection that considers the content angle — not just the keyword.

**Content Angle Comparison:**

Two content ideas may target different keywords but produce nearly identical content:

| Content Idea A                                | Content Idea B                                | Keyword Match  | Content Angle Match | Verdict            |
|-----------------------------------------------|-----------------------------------------------|----------------|---------------------|--------------------|
| "Best CRM for Startups"                       | "Top CRM Tools for Early-Stage Companies"     | YES (semantic) | YES                 | CONTENT DUPLICATE  |
| "How to Set Up HubSpot CRM"                   | "HubSpot CRM Setup Guide"                     | YES (semantic) | YES                 | CONTENT DUPLICATE  |
| "HubSpot vs Salesforce"                       | "Salesforce vs HubSpot Comparison"            | YES (semantic) | YES                 | CONTENT DUPLICATE  |
| "Best CRM for Real Estate Agents"             | "Best CRM for Healthcare Providers"           | NO             | NO (different niche) | NOT DUPLICATE     |
| "CRM Integration with Slack"                  | "CRM Integration with Gmail"                  | NO             | NO (different integration) | NOT DUPLICATE |

**Content Angle Extraction:**

For each content idea:
- Subject: What/who the content is about.
- Angle: How the content approaches the subject (tutorial, comparison, listicle, case study, definition).
- Context: Any specific modifier (audience, tool, use case).

Content ideas are duplicates when Subject AND Angle match — regardless of keyword phrasing differences.

### ▸ CANNIBALIZATION DETECTION — ADVANCED IMPLEMENTATION

Beyond basic cannibalization risk assessment in Step 5, the advanced cannibalization detection maps the full competitive overlap between new candidates and the existing content landscape.

**Cannibalization Impact Score:**

For each detected cannibalization risk, score the severity:

| Factor                                                                          | Weight | Score Range |
|---------------------------------------------------------------------------------|--------|-------------|
| Same primary keyword (exact or semantic)                                       | 40%    | 0–10        |
| Same intent (TOFU/MOFU/BOFU)                                                   | 30%    | 0–10        |
| Same cluster                                                                    | 20%    | 0–10        |
| Same target audience                                                            | 10%    | 0–10        |

`Cannibalization_Score = (factor_score × weight) for each factor, summed`

| Score Range | Cannibalization Level | Action                        |
|-------------|----------------------|-------------------------------|
| 8–10        | CRITICAL             | REJECT — do not proceed       |
| 5–7         | HIGH                 | REJECT with CANNIBALIZATION flag |
| 3–4         | MEDIUM               | REVIEW — may be differentiated |
| 0–2         | LOW                  | KEEP with CANNIBALIZATION WATCH tag |

## SECTION 6 — DEDUPLICATION RULES
## ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

### System-Level Deduplication Rules

These rules govern the deduplication engine's own behavior — they are the meta-rules of the system.

**Rule 1 — Never Allow the Same Intent Twice (in the same cluster)**

No two keywords assigned to the same cluster may serve the same user intent at the same funnel stage. Two MOFU keywords for the same core concept in the same cluster = intent duplicate. One is rejected regardless of how different their phrasings are.

**Rule 2 — Never Allow the Same Primary Keyword Twice (anywhere)**

No two content pieces in the entire system (published, planned, or draft) may share the same primary keyword. The primary keyword is an exclusive assignment. This rule is enforced at the site level — not just the cluster level.

**Rule 3 — Never Allow Pipeline Duplication**

If a keyword is already the target of a planned or draft piece, it cannot be recommended as a new creation target. The pipeline is treated as equivalent to published content for deduplication purposes.

**Rule 4 — Always Check Memory**

Every deduplication run must query memory before producing its clean list. Memory is checked at Step 1 (for exact matches in published/memory DB) AND at Step 7 (for full cross-session deduplication). Skipping the memory check is a protocol violation.

**Rule 5 — Rejection Log Is Permanent**

The rejection log from every deduplication run is stored in `memory-controller.skill` permanently. It is not overwritten by subsequent runs. Rejection logs accumulate as an audit trail — they are used in Step 7 (cross-session dedup) to prevent the same rejections from being re-evaluated repeatedly.

**Rule 6 — Failsafe: Keep When Uncertain**

When the deduplication engine cannot make a confident determination (ambiguous semantic match, uncertain memory record, conflicting signals), it applies the failsafe: KEEP the candidate, assign confidence = LOW, flag as REVIEW NEEDED. The rejection of a valid unique keyword is a worse outcome than keeping a potential near-duplicate.

**Rule 7 — Deduplication Is Applied to ALL Types**

Deduplication applies equally to: single keywords, keyword phrases, long-tail variants, LSI keywords, content titles, content plans, cluster keyword assignments, and roadmap entries. No content type is exempt.

## SECTION 7 — ANTI-FLUFF RULES
## ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

### Prohibited Deduplication Behaviors

| Prohibited Behavior                                                                            | Reason                                                                              |
|------------------------------------------------------------------------------------------------|-------------------------------------------------------------------------------------|
| Rejecting a keyword without logging a specific reason                                          | Every rejection must be auditable — blind removal is a quality control failure     |
| Using `"too similar"` as a rejection reason without specifying which signal matched           | Vague reasoning cannot be verified, reversed, or appealed by the user              |
| Removing items because they "seem redundant" without applying the 5-signal semantic test      | Intuitive removals may remove strategically distinct keywords                      |
| Applying STRICT mode to an exploratory ideation session                                        | Over-deduplication at ideation stage kills creative breadth                        |
| Rejecting a REFRESH CANDIDATE as a simple rejection without running Step 6                    | Existing thin content IS a gap — it should be updated, not blocked                |
| Flagging a LOW-confidence match as a definitive REJECT                                         | Low-confidence matches go to REVIEW — they are never auto-rejected                |
| Rejecting a cross-cluster keyword without first checking if it should be reference-tagged     | Cluster deduplication should tag, not always reject                               |
| Running deduplication without loading the memory database                                      | Cross-session deduplication is mandatory — not optional                           |
| Returning only the clean list without the rejection log                                         | Both outputs are mandatory — the rejection log is the audit trail                 |
| Allowing a calling skill to bypass deduplication with a flag or parameter                     | There is no bypass. Deduplication is mandatory for all generation flows.           |

### Required Deduplication Behaviors

- Every rejection must cite: the rejection type, the matching reference, and the step that triggered it.
- Every REVIEW-flagged item must explain specifically what makes it uncertain.
- The rejection log must be stored in memory after every run.
- The clean list must include confidence levels for every item.
- Refresh candidates must be separated from both the clean list and the rejection log.

## APPENDIX B — DEDUPLICATION STEP SEQUENCE QUICK REFERENCE
## ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

```
DEDUPLICATION ENGINE — STEP SEQUENCE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
INPUT: Candidate pool + Reference DBs + Memory DB

STEP 1 — EXACT DEDUPLICATION
  → Normalize all candidates
  → Remove exact matches within batch
  → Remove exact matches vs. reference DBs
  → Reject: EXACT DUPLICATE

STEP 2 — SEMANTIC DEDUPLICATION
  → Apply 5-signal semantic equivalence test
  → Detect synonym, word-order, question/statement, plural, qualifier variants
  → Keep: more specific / question form / current-year variant
  → Reject: SEMANTIC DUPLICATE (N/5 signals)

STEP 3 — INTENT DEDUPLICATION
  → Extract intent for each surviving candidate
  → Compare intent pairs within same cluster
  → Keep: clearer, more specific expression of the goal
  → Reject: INTENT DUPLICATE

STEP 4 — CLUSTER DEDUPLICATION
  → Check cluster membership for each candidate
  → Detect cross-cluster keyword conflicts
  → Apply reference tagging for genuinely cross-cluster keywords
  → Reject: CLUSTER DUPLICATE (or TAG: CROSS-REFERENCE)

STEP 5 — PIPELINE AWARENESS
  → Cross-reference vs. published content (primary + secondary keywords)
  → Cross-reference vs. draft and planned content
  → Assess cannibalization risk
  → Reject: PUBLISHED / PIPELINE CONFLICT / CANNIBALIZATION RISK
  → Exception: Route to Step 6 if existing content qualifies for refresh

STEP 6 — REFRESH EXCEPTION
  → Evaluate rejected items from Step 5 against refresh criteria
  → Apply freshness + coverage score + SERP performance matrix
  → Output: REFRESH CANDIDATE (not rejected; not in clean list)

STEP 7 — CROSS-SESSION DEDUPLICATION
  → Query memory for prior-session keywords, rejections, assignments
  → Apply memory exception protocol for voided rejections
  → Reject: MEMORY DUPLICATE (EXACT / SEMANTIC)
  → Keep: MEMORY EXCEPTION (with flag)
  → Keep: ABANDONED (with VERIFY flag)

STEP 8 — FINAL OUTPUT
  → Assign final status: CLEAN / REVIEW / REFRESH
  → Assign confidence: HIGH / MEDIUM / LOW
  → Write rejection log to memory
  → Update keyword-memory dedup statuses
  → Return: Clean List + Rejection Log to calling skill

FAILSAFE AT EVERY STEP:
  Uncertain? → KEEP. Flag as REVIEW NEEDED. Log uncertainty.
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

REJECTION REASON CODES — COMPLETE REFERENCE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Step 1 Codes (Exact):
  EXACT_BATCH       : Identical to another candidate in the same input batch
  EXACT_PUBLISHED   : Exact match to a published page's primary keyword
  EXACT_SECONDARY   : Exact match to a published page's secondary keyword (lower severity)
  EXACT_PIPELINE    : Exact match to a planned/draft piece's target keyword
  EXACT_MEMORY      : Exact match to a prior-session keyword in memory
  EXACT_URL         : Content idea slug duplicates an existing URL

Step 2 Codes (Semantic):
  SEM_SYNONYM       : Synonym substitution — same meaning, different word
  SEM_WORDORDER     : Word order reversal — same words, different arrangement
  SEM_QUESTION      : Question vs. statement form of same content
  SEM_PLURAL        : Plural/singular variant of same keyword
  SEM_QUALIFIER     : With/without trailing format qualifier (guide, tips, etc.)
  SEM_YEAR          : Same keyword, different year variant
  SEM_MATCH         : General semantic equivalence (5-signal test, N/5 signals)

Step 3 Codes (Intent):
  INTENT_DUP        : Same user goal at same funnel stage as kept keyword
  INTENT_VAGUE      : Vague variation of a clearer keyword with identical intent
  INTENT_CLUSTER    : Grouped into intent cluster; non-representative candidate

Step 4 Codes (Cluster):
  CLUSTER_DUP       : Keyword exists in another cluster — cross-cluster duplicate
  CLUSTER_ASSIGNED  : Keyword already assigned to a cluster in keyword-memory
  CLUSTER_TAG       : Not rejected — applied cross-reference tag instead

Step 5 Codes (Pipeline):
  PUBLISHED_PK      : Already covered as primary keyword at [URL]
  PUBLISHED_SK      : Already covered as secondary keyword at [URL]
  PUBLISHED_TOPIC   : Topic equivalence with existing published content
  PIPELINE_DRAFT    : Already in draft status as [title]
  PIPELINE_PLANNED  : Already in planned status as [title]
  CANNIBALIZATION   : Would compete with [URL] for same SERP position [score]

Step 6 Codes (Refresh — not rejection):
  REFRESH_AGE       : Existing content is STALE/OUTDATED — refresh recommended
  REFRESH_THIN      : Existing content has coverage score 1–2 — expansion needed
  REFRESH_SERP      : Existing content underperforms SERP — rewrite recommended
  REFRESH_COMBINED  : Multiple refresh triggers — comprehensive update needed

Step 7 Codes (Memory):
  MEM_EXACT         : Exact match to prior-session keyword — already generated
  MEM_SEMANTIC      : Semantic match to prior-session keyword — same intent used
  MEM_ASSIGNED      : Keyword was assigned to planned content in prior session
  MEM_EXCEPTION     : Prior rejection voided — reason no longer valid
  MEM_ABANDONED     : Previously abandoned — kept with verify flag

Status Codes (Final):
  CLEAN             : Passed all checks — ready for use
  REVIEW            : Passed with borderline flags — human check recommended
  REFRESH           : Matched existing content — update, don't create new
